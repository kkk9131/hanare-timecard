## 検証済みフロー

1. `GET /api/system/health` → `{status: "ok", ok: true}`
2. `POST /api/auth/admin-login` (oyakata / hanare2026) → Set-Cookie
3. `POST /api/employees` (staff "E2E 太郎", PIN 4242) → 201 / employee.id
4. `POST /api/auth/pin-login` (作成した staff) → Set-Cookie
5. `POST /api/punches` clock_in → `next_state: working`
6. `GET /api/punches/me/state` → `state: working`
7. clock_in の punched_at を SQL で 90 分前に backdate
   (状態遷移は「最後の punch_type」のみ見るので安全)
8. `POST /api/punches` clock_out → `next_state: off`
9. `GET /api/punches/me/summary` → `worked > 0` を確認
10. `POST /api/shifts` → 201 / shift.id > 0
11. `POST /api/corrections`
    (`target_date`, `requested_type: clock_in`, `requested_value: ms`, reason) → 201
12. `POST /api/corrections/:id/approve` → `status: approved`
13. `GET /api/audit?limit=100` → `logs.length > 0`, `employee.create` を含む
14. `GET /api/exports/period.xlsx?from=&to=&store_id=1`
    - Content-Type が xlsx MIME
    - Buffer.length > 0
    - 先頭 2 バイト `PK` (ZIP magic) で妥当性確認

## ハマりどころ (記録)

- `approveCorrection()` は `target_punch_id` か
  `requested_value + requested_type` のいずれかが無いと
  `missing_value` (422) で落ちる。初版は reason だけ渡して落ちたため、
  `requested_type: clock_in` と `requested_value` (unix ms) を明示追加した
- `createPunch` は直後に clock_in → clock_out を投げると worked_minutes が
  0 分になり `summary.worked > 0` が満たせない。SQL で clock_in punch の
  `punched_at` を 90 分前に書き換えることで解決 (状態遷移判定は最後の
  punch_type しか見ないため副作用なし)
