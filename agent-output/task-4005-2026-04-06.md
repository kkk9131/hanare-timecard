## 主要判断

- **store_id フィルタの解釈**: api-spec.md と data-model.md で `correction_requests` 自体には store_id がないため、`employee_stores` を経由して申請者の所属店舗で絞り込む方式を採用
- **打刻漏れ申請時の store_id**: 申請者の primary store（なければ assigned 店舗の先頭）を使用
- **note の上書き**: 既存 note を破棄せず ` / correction#<id>` を appended する形で保持
- **audit_logs の entity**: 承認時は `time_punch`、却下時は `correction_request` を entity_type に設定（実際に変更されたエンティティに合わせる）
- **service 単位の責務分離**: HTTP 層では HTTPException ではなく result kind を返し、route 層でステータスコードに変換（既存 punches/shifts と統一）
