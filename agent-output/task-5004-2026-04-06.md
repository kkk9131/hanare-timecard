## design direction

ticket は "AppShell を使った左サイドバーナビ + WashiCard 内に大きく集計値" を要求。
既存の wa-modern token (sumi/washi/shu/kincha) と既存コンポーネント (WashiCard, SumiButton, StatePill, AppShell) を踏襲し、
restrained editorial レイアウトを採用:

- 当月集計 (E01) は 4 列の hairline grid + 明朝 tabular-nums の大数字 + 漢字 1 字 (実/残/休/深) を半透明で背景に置く eyebrow scheme
- テーブル (E02/E03/E05) は枠を排し、点線下線 + 金茶ヘッドラインのみで余白を活かす
- フォームは sumi-850 背景 + kincha-400 focus ring (2px) で和紙ライク
- E04 月ビューは 7×6 hairline グリッド、当日は kincha-300 ハイライト
- 左サイドバー: ロゴ + 5 項目 (ダッシュボード / 履歴 / 修正申請 / シフト / 希望) + ログアウトボタン
- 各ナビ項目には漢字 1 字グリフ (今/履/正/表/願) を 28×28 ヘアライン枠で添える

`/frontend-design:frontend-design` skill のガイドラインに沿い、generic な「Inter + 角丸 + 紫グラデ」を完全に避け、
spec.md UI/UX 方針 (墨黒 + 和紙 + 朱 + 金茶 / Noto Serif JP) に強く合わせた。

## known_gaps

- 修正申請フォームは「対象日 + 修正後時刻 + 種別 + 理由」のみ受け付け、`target_punch_id` には対応しない
  (一覧から特定打刻を選ぶ UI を後続チケットで足す想定)。spec / ticket には punch_id 連結は必須要件として書かれていない。
- 通知 (修正申請ステータス) は最新 5 件を E01 に直接表示するシンプル実装。本物の "未読/既読" 管理はしない
  (spec の 3.4 にも "通知" の細目は無く、unread state を持つテーブルが未定義)。
- E01 のシフト「店舗 #N」表記は store master の name を取れていない。task-4001 系の stores API を呼ぶ
  ことで改善可能だが、shift カードに描画するだけなら最小実装で済むので保留した。
- pre-existing: `src/client/routes/AdminShifts.tsx` に構文エラー (JSX 閉じタグ漏れ等) があり tsc はそこで
  失敗する。task-5004 の scope 外なので触らない。`vite build` (esbuild) は通る。
- E01 の「今月の勤務」サマリ値はサーバ `/api/punches/me/summary` の戻り値をそのまま信用している。
  サーバ側で 0 fallback 実装になっている場合は表示も 0 になる。
