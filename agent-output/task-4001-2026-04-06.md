- `src/server/routes/system.ts`: `GET /health` を定義（`/api/system/health` に mount）
- `src/server/app.ts`: `createApp()` ファクトリ
  - `logger()` ミドルウェア
  - `cors({ origin: 反射, credentials: true })` (LAN + Vite dev 5173 想定)
  - `sessionMiddleware` を全ルートに適用
  - `/api` サブアプリに `systemRoutes` を mount し、追加で `/api/health` を bc パス用に直接登録
  - `NODE_ENV=production` の時のみ `serveStatic({ root: './dist/client' })` + SPA フォールバックで `dist/client/index.html` を返す（dev は Vite が担当）
  - `notFoundHandler` / `errorHandler` を接続
- `src/server/index.ts`: `serve({ fetch: app.fetch, hostname: '0.0.0.0', port })` で起動。`PORT` 環境変数対応 (default 3000)、起動時に `os.networkInterfaces()` から LAN IPv4 を列挙しバナー出力

## known_gaps

- `migrate.ts` を起動時に自動実行する方針（チケット本文の「起動時に migrate.ts を内部で呼ぶ」）は今回未実装。現状は外部で `npm run migrate` を事前実行する前提。`sessions` テーブル自体はマイグレーション済みの環境で動作検証済み。Orchestrator 判断で別チケットに分離 or 後続で追加する必要がある
- `lib/crypto.ts` の `sign`/`unsign` は現状どこからも呼ばれていない。セッション ID 自体はランダム値で DB 検証するため署名不要だが、後続の CSRF トークン等に備えて残してある
- Cookie 署名ではなくランダム ID + DB 検証方式にしたため、HMAC 署名は将来オプション。`HANARE_SESSION_SECRET` 環境変数も `sign`/`unsign` 経由でのみ使われる
- `serveStatic` のフォールバックは `dist/client/index.html` 単発で、History API 用の真の SPA フォールバックはビルド時確認が必要（dist 未生成のため未検証）。client ビルドが揃う phase で結合確認すべき
- 起動時のポート衝突ハンドリング（`EADDRINUSE`）は `@hono/node-server` 任せで、フレンドリーエラーは未実装
- ロガーは `hono/logger` をそのまま使用。構造化ログや request id は未実装
