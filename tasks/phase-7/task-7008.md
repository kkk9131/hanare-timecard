# 修正申請の店舗指定を明示化

## 背景

PR #3 で、既存打刻に紐づく修正申請は `time_punches.store_id` を基準に店舗スコープを判定するようにした。
一方、`target_punch_id` がない新規打刻申請は、現行スキーマに `store_id` がないため、従業員の所属店舗を基準に扱っている。

この仕様でも既存運用は壊れないが、複数店舗所属の従業員が新規打刻申請を出す場合、どの店舗の申請なのかをデータだけで明確にできない。

## What

- `correction_requests` に申請対象店舗を明示できるようにする
- `POST /api/corrections` で `store_id` を受け取れるようにする
- `target_punch_id` がある場合は、対象打刻の店舗と `store_id` の整合性を確認する
- `target_punch_id` がない場合は、指定された `store_id` に従業員が所属していることを確認する
- 修正申請一覧・承認・却下は、明示された店舗を優先して権限判定する

## Why

店舗を明示しておくと、店長が確認すべき申請がより正確になる。
あとから監査ログや月次エクスポートを見返すときも、「どの店舗の打刻修正だったか」を非エンジニアでも追いやすくなる。

## How

- DB マイグレーションで `correction_requests.store_id` を追加する
- 既存データは次の優先順位で `store_id` を補完する
  - `target_punch_id` がある場合: 対象打刻の `store_id`
  - `target_punch_id` がない場合: 従業員の primary store
- zod schema / shared type / API spec を更新する
- `createCorrection` / `listCorrections` / 承認・却下前チェックを `store_id` 基準に寄せる
- 複数店舗所属の従業員で、新規打刻申請の店舗境界テストを追加する

## Done When

- `target_punch_id` がない修正申請でも、対象店舗が DB に保存される
- manager は自店舗の新規打刻申請だけ一覧表示・承認・却下できる
- manager は他店舗の `store_id` を指定した新規打刻申請を扱えない
- 既存の修正申請データに `store_id` が補完される
- API ドキュメントに `store_id` の扱いが記載される
- `npm test` が pass する
