# 初回オンボーディング

雀庵 タイムカードを本番で使い始める前に、店主・店長・現場担当者が一緒に確認するための初回ガイド。

目的は、初日に「どの画面を開けばよいか」「誰が何をできるか」「月末と復旧で何をすればよいか」を迷わない状態にすること。

## 全体像

2 店舗の勤怠は、1 台の本番サーバー PC に集める。

```text
本店の打刻端末
  └─ ブラウザでアクセス
                      ┌─ data/hanare.db
本番サーバー PC ──────┤
                      └─ data/backups/
離れの打刻端末
  └─ ブラウザでアクセス

月末: 管理者が `/admin/exports` から Excel を出力
```

各端末にアプリを別々に入れるのではなく、各端末のブラウザから同じサーバー PC を見る。

## 0. 事前に用意するもの

- 本番サーバー PC 1 台
- 本店・離れで使う打刻端末
- 管理者ログイン ID / パスワード
- 店長ログイン ID / パスワード
- 従業員一覧と所属店舗
- バックアップを保管する外付けディスク、別 PC、または共有フォルダ

## 1. 最初に確認する URL

本番サーバー PC で `./start.sh` を起動すると、画面に URL が表示される。

```text
hanare-timecard server
  -> http://localhost:3000
  -> http://192.168.1.42:3000  (LAN)
```

使い分け:

| 開く場所 | URL |
| --- | --- |
| 本番サーバー PC | `http://localhost:3000` |
| 本店・離れの打刻端末 | `http://<表示されたIP>:3000` |
| 管理者ログイン | `/admin/login` |
| 管理ダッシュボード | `/admin` |
| 月次エクスポート | `/admin/exports` |

## 2. 権限の違いを確認する

| 役割 | できること | 開ける管理画面 |
| --- | --- | --- |
| 従業員 (`staff`) | 打刻、自分の履歴確認、修正申請、公開シフト確認 | 管理画面は開けない |
| 店長 (`manager`) | 自店舗の日次確認、シフト編成、修正申請の承認/却下 | `/admin`, `/admin/shifts`, `/admin/corrections` |
| 管理者 (`admin`) | 全店舗の管理、従業員/店舗マスタ、月次エクスポート、監査ログ、バックアップ | `/admin` 配下の全画面 |

確認ポイント:

- 店長でログインすると、従業員・店舗・エクスポート・監査ログは表示されない
- 店長で `/admin/exports` を直接開くと、403「権限がありません」と表示される
- 管理者でログインすると、すべての管理メニューが表示される

## 3. 初回セットアップ

管理者が行う。

1. 打刻トップ右上の「管理者画面へ」を押す、または `/admin/login` を開く
2. 管理者 ID / パスワードでログインする
3. `/admin/employees` を開く
4. 管理者と店長の初期パスワードを本番用に変更する
5. 従業員名、退職者、所属店舗を確認する
6. 兼務スタッフが本店・離れの正しい店舗に所属しているか確認する
7. `/admin/stores` で店舗名が現場で分かる表記になっているか確認する

初期データ投入用の `npm run seed` は初回セットアップ時だけ使う。実運用開始後に実行すると、既存データに影響するため使わない。

## 4. 打刻の練習

本店と離れで、それぞれ 1 回ずつ確認する。

1. 打刻端末で `http://<サーバーPCのIP>:3000` を開く
2. 店舗切り替えで対象店舗を選ぶ
3. テスト対象の従業員名を押す
4. `出勤する` を押す
5. 完了画面が出て、数秒後に打刻トップへ戻ることを確認する
6. 必要に応じて `退勤する` まで実施する

確認ポイント:

- 本店の端末と離れの端末が、同じ本番サーバー PC を見ている
- 従業員が所属していない店舗では打刻できない
- 打刻後に `/me/history` で履歴を確認できる

## 5. 修正申請の練習

打刻漏れや間違いが起きた時の練習を 1 回行う。

従業員側:

1. `/me/history` を開く
2. 修正したい打刻から修正申請を作る
3. 修正後の時刻と理由を入力して送信する

店長または管理者側:

1. `/admin/login` からログインする
2. `/admin` の未処理申請を確認する
3. `/admin/corrections` で申請内容を見る
4. 内容が正しければ承認する
5. 却下する場合は理由を入力する

承認後は、月次エクスポートにも修正後の内容が反映される。

## 6. シフト公開の練習

店長または管理者が行う。

1. `/admin/shifts` を開く
2. 週ビューまたは月ビューでシフトを入力する
3. 重複や人員不足の表示がないか確認する
4. 公開する
5. 従業員側の `/me/shifts` で公開シフトが見えることを確認する

店長は自店舗のシフトを扱う。管理者は全店舗を確認できる。

## 7. 月次エクスポートの練習

管理者が行う。店長アカウントでは実行できない。

1. `/admin/login` から管理者でログインする
2. `/admin/exports` を開く
3. 店舗を「本店」「離れ」「全店舗」から選ぶ
4. 期間を指定する
5. 通常は `.xlsx` を選び、「Excel で書き出す」を押す
6. 当月分をすぐ出す場合は「今月をエクスポート」を押す
7. ダウンロードした Excel を開く

確認ポイント:

- 氏名と店舗が正しい
- 出勤・退勤時刻が入っている
- 休憩時間、実働時間、深夜時間が表示されている
- 承認済みの修正申請が反映されている
- 日本語が文字化けしていない

## 8. バックアップ確認

本番前に必ず 1 回確認する。

```bash
npm run backup
```

確認すること:

- `data/backups/` に `hanare-YYYYMMDD-HHMMSS.db` 形式のファイルができる
- バックアップファイルを外付けディスク、別 PC、または共有フォルダへコピーできる
- 担当者が `docs/operations.md` の復旧手順を開ける

## 9. 復旧手順の読み合わせ

復旧の基本は、壊れた可能性がある DB を残してから、バックアップを戻すこと。

```text
アプリ停止
  ↓
最新バックアップ確認
  ↓
現在の data/hanare.db を data/hanare.db.broken に変更
  ↓
バックアップを data/hanare.db としてコピー
  ↓
./start.sh で再起動
```

実際のコマンドは [運用ガイド](operations.md) の「DB 破損 / 復旧」を参照する。

## 10. 初回オンボーディング完了チェック

- [ ] 本番サーバー PC を 1 台に決めた
- [ ] 本店・離れの端末から同じ URL を開ける
- [ ] 管理者と店長の初期パスワードを変更した
- [ ] 従業員マスタと所属店舗を確認した
- [ ] 店長で見える画面が `/admin`, `/admin/shifts`, `/admin/corrections` のみである
- [ ] 店長で `/admin/exports` を開くと 403「権限がありません」になる
- [ ] 管理者で `/admin/exports` から Excel を出せる
- [ ] 本店・離れでテスト打刻できる
- [ ] 修正申請を作成し、承認できる
- [ ] シフトを作成し、公開できる
- [ ] `npm run backup` でバックアップが作成される
- [ ] 復旧手順の場所を担当者が知っている
