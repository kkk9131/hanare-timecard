# hanare-timecard 仕様書

## 1. プロジェクト概要・目的

鶏肉専門店「雀庵 本店」および「雀庵 離れ」の 2 店舗で共用する、シフト管理機能付きタイムカード web アプリケーション。紙のタイムカードと手書きシフト表の運用から脱却し、打刻・集計・シフト調整・給与計算用データ出力までを一元化する。クラウド不要でローカル 1 台のマシン（店舗 PC or 店主 Mac）上で完結して動作することを前提とし、非エンジニアの従業員が迷わず使える和モダンな UI を備える。

### ゴール

- 打刻の正確性と改ざん耐性を担保する
- 月次の勤怠集計を Excel に流し込むまでの手間を最小化する
- 店主/管理者がシフトを直感的に編成できる
- 店舗ブランド「雀庵」の世界観に合致したインターフェース

## 2. ターゲットユーザーと利用シーン

### 従業員（アルバイト/パート/社員）

- 出勤時・退勤時・休憩時に共用端末（店舗の iPad or PC）で打刻
- 自分の打刻履歴・今月の勤務時間を確認
- 打刻漏れや誤打刻を「修正申請」として店長に提出
- 翌週・翌月のシフトを閲覧

### 店長（各店舗の責任者）

- シフトをドラッグ操作などで編成・公開
- 従業員の修正申請を承認/却下
- その店舗の勤怠状況をダッシュボードで確認

### 管理者（オーナー/本部）

- 両店舗を横断して従業員・シフト・勤怠を管理
- 月次締め処理と Excel/CSV エクスポート
- 従業員マスタ管理（入社・退社・時給設定）

### 主な利用シーン

- 出勤時: 共用端末のトップ画面で自分の名前を選ぶ → 出勤打刻
- 休憩開始/終了: 同様に氏名選択後に 1 タップ
- 月末: 管理者が締めて xlsx エクスポート → 既存の給与 Excel に貼り付け
- 週末: 店長が翌週シフトを編成して公開 → 従業員に通知（アプリ内）

## 3. 主要機能一覧

### 3.1 認証・店舗切替

- 従業員ログイン: 従業員選択のみ（共用端末向け簡易認証）
- 管理者/店長ログイン: ID + パスワード
- 店舗コンテキスト: ログイン後の画面は「本店」「離れ」いずれかに属する。兼務従業員は店舗を切替可能
- セッションは共用端末想定で短時間（例: 打刻後即ログアウト、管理画面は数時間）

### 3.2 タイムカード機能

- 出勤 / 退勤 / 休憩開始 / 休憩終了 の 4 打刻
- 打刻時に端末時刻ではなくサーバ時刻（ローカル DB ホスト時刻）を記録
- 連続同種打刻の防止（例: 出勤中にさらに出勤を押せない）
- 自分の打刻履歴閲覧（日/週/月）
- 当月の累計勤務時間、残業時間、休憩時間の表示
- 修正申請: 対象打刻 or 対象日を指定し、修正後の時刻と理由を記入して提出

### 3.3 シフト管理ダッシュボード（管理者・店長）

- 週ビュー/月ビューでのシフト表示
- 従業員 × 日付のセルに勤務時間帯を割り当て
- ドラフト状態と公開状態を区別
- 公開後、従業員側からも閲覧可能に
- シフト希望提出機能（従業員が出勤可能日時を申告 → 店長が参照）
- 人員不足日のハイライト

### 3.4 修正申請ワークフロー

- 従業員: 申請作成 → 店長に通知
- 店長/管理者: 承認 or 却下（却下理由必須）
- 承認されたら打刻データを上書きし、監査ログに旧値を残す

### 3.5 エクスポート

- 期間指定で従業員別勤怠サマリ（氏名・店舗・日別打刻・総労働時間・残業時間・深夜時間・休憩時間）
- CSV（UTF-8 BOM 付き、Excel 直開き互換）
- xlsx（シート 1 枚、ヘッダー固定行、列幅自動調整）
- 店舗単位 / 全店舗 両方選択可能

### 3.6 従業員・店舗マスタ管理（管理者）

- 従業員の追加/編集/退職処理
- 所属店舗（複数可）、時給、権限ロール、管理者ログイン情報の設定
- 店舗情報（名称、営業時間、定休日）

### 3.7 監査ログ

- 打刻・修正・承認・マスタ変更を時系列で記録（管理者のみ閲覧）

## 4. 受け入れ条件

### 認証

- [ ] 従業員は自分の名前を選択して 3 秒以内に打刻画面へ進める
- [ ] 打刻は PIN 入力なしで出勤 / 退勤 / 休憩開始 / 休憩終了を記録できる
- [ ] 管理者ログインは ID + パスワードで、セッションは少なくとも 2 時間有効

### タイムカード

- [ ] 出勤中の従業員は「退勤」「休憩開始」のみ押せ、「出勤」は disabled になる
- [ ] 打刻後、画面に「◯◯さん、お疲れさまです。出勤を記録しました（HH:MM）」と和モダンな表現で結果を表示する
- [ ] 打刻データは DB に保存され、アプリを再起動しても失われない
- [ ] 従業員は自分の過去 3 ヶ月分の打刻履歴を閲覧できる
- [ ] 日をまたぐ勤務（深夜営業）も 1 勤務として集計される

### シフト

- [ ] 店長は 1 週間分のシフトを 5 分以内に編成・公開できる
- [ ] 従業員は公開済みシフトのみ閲覧でき、ドラフトは見えない
- [ ] 同一従業員が同日に重複する時間帯を割り当てられない
- [ ] シフト公開後の変更は履歴として残る

### エクスポート

- [ ] 管理者は月末に 1 クリックで当月分 xlsx をダウンロードできる
- [ ] エクスポートされた xlsx を Excel for Mac / Excel for Windows で開いて文字化けしない
- [ ] CSV は UTF-8 BOM 付きで Excel にダブルクリック展開して日本語が崩れない
- [ ] 集計値（総労働時間等）は手計算と 1 分未満の誤差で一致する

### 修正申請

- [ ] 従業員が申請を送ると店長のダッシュボードに未処理件数が表示される
- [ ] 承認された修正は打刻データに反映され、監査ログに旧値と新値が残る

### 店舗区別

- [ ] 従業員一覧・シフト・勤怠集計はすべて店舗フィルタで「本店」「離れ」「全店舗」を切替可能
- [ ] 兼務従業員はどちらの店舗のシフトにも載せられる

## 5. 非機能要件

### 動作環境

- ローカル 1 台のマシン（macOS arm64 / Windows も動けばベター）で起動可能
- インターネット接続がなくても機能する（外部 API 依存禁止）
- 同一 LAN 内の複数端末（店舗 iPad など）から同一サーバにアクセスできること

### データ保存

- 組み込み DB（SQLite 等）を推奨。単一ファイルで完結
- データベースファイルは明示的なパスに置かれ、バックアップ/コピーが容易
- 日次自動バックアップ（同ディレクトリ内にタイムスタンプ付きコピー、直近 30 世代保持）

### Excel 互換

- xlsx は ISO/IEC 29500 準拠。Excel 2016 以降で開けること
- CSV は UTF-8 BOM + CRLF

### パフォーマンス

- 打刻操作は 500ms 以内にフィードバックを返す
- 月次集計 1 店舗 30 名分で 2 秒以内

### セキュリティ

- 管理者 / 店長パスワードはハッシュ化保存（bcrypt 等）
- 監査ログは削除不可（UI 経由では）
- ローカル LAN 前提のため TLS は必須としないが、構成で有効化できること

### レスポンシブ

- タブレット縦/横、デスクトップに対応
- 打刻画面は特に iPad 縦 (768×1024) で最適化

### 可搬性

- `npm install && npm run start` 相当の 2 コマンドで起動完了
- DB ファイルを別マシンにコピーすれば環境移行できる

## 6. UI/UX 方針

### ブランドコンセプト

- 店舗「雀庵 本店 / 雀庵 離れ」は京町家風・和モダン・鶏肉専門店
- キーワード: 墨黒、焦茶の木目、障子の白、暖色の行灯、朱の差し色、余白、縦書き的構成

### ビジュアル方向

- **ベース**: 墨黒 (#111) または濃紺に近い漆黒を基調とした暗めの画面
- **アクセント**: 和紙のような温かい白 (#F5EFE4)、朱赤 (#B23A2E)、金茶 (#B8860B) を差し色
- **木目テクスチャ**: サブトルなノイズ/木目をカード背景に
- **タイポグラフィ**: 見出しは明朝系（游明朝、Noto Serif JP、しっぽり明朝など）縦書き活用。本文は游ゴシック or Noto Sans JP Light
- **ロゴ表示**: 「雀庵」漢字を縦組みで左上に配置。店舗切替で「本店／離れ」のサブタイトルが入れ替わる
- **アイコン**: 線画、筆書き風
- **アニメーション**: 障子が開くような横スライド、行灯の明滅を模したホバー。過剰な動きは避ける

### UX 方針（非エンジニア配慮）

- 打刻トップ画面はボタン 1〜2 個のみ。文字は大きく、現在時刻を大きく表示
- 専門用語を使わない（「クロックイン」ではなく「出勤する」）
- エラーメッセージは丁寧語でわかりやすく表示する
- 確認ダイアログは「本当に退勤しますか？ (HH:MM)」のように文脈を明示
- タップ対象は最小 44×44pt 以上
- 色だけに頼らない状態表現（アイコン + テキスト併記）
- ダークモード前提で可読性を最優先にコントラスト確保（WCAG AA 相当）

## 7. データモデル概要

### Store（店舗）

- id, name (雀庵 / 雀庵 離れ), display_name, opening_time, closing_time, created_at

### Employee（従業員）

- id, name, kana, role (staff/manager/admin), hourly_wage, hire_date, retire_date (nullable), note

### EmployeeStore（従業員と店舗の多対多、兼務対応）

- employee_id, store_id, is_primary

### TimePunch（打刻）

- id, employee_id, store_id, punch_type (clock_in/clock_out/break_start/break_end), punched_at, source (kiosk/admin), note

### WorkDay（日次集計、任意キャッシュ）

- employee_id, store_id, date, worked_minutes, break_minutes, overtime_minutes, night_minutes

### Shift（シフト）

- id, employee_id, store_id, date, start_time, end_time, status (draft/published), created_by, created_at

### ShiftRequest（シフト希望）

- id, employee_id, date, time_range, note, submitted_at

### CorrectionRequest（修正申請）

- id, employee_id, target_punch_id (nullable), target_date, requested_at, requested_value, reason, status (pending/approved/rejected), reviewer_id, reviewed_at, review_comment

### AuditLog（監査ログ）

- id, actor_id, action, entity_type, entity_id, before_json, after_json, occurred_at

### User（管理者ログイン用、Employee と別 or 併合可）

- 実装時に Employee.role で統合するか別テーブルにするか architect が決定

## 8. スコープ外（今回作らないもの）

- 給与計算の自動化（計算は Excel 側で継続）
- 税・社会保険計算
- POS / 売上管理連携
- 顔認証 / 指紋認証打刻
- スマートフォン従業員向けネイティブアプリ
- クラウド同期・マルチテナント
- メール/SMS/LINE 通知（アプリ内通知のみ）
- 多言語対応（日本語のみ）
- 勤務スケジュールの AI 自動最適化

## 9. 推奨技術スタック候補

ローカル簡潔運用と非エンジニア運用を前提に、複数候補を提示する。最終決定は architect フェーズで行う。

### 候補 A: Node.js 単一プロセス（推奨度: 高）

- ランタイム: Node v25（インストール済み）
- フレームワーク: SvelteKit or Next.js（SSR + API 同居）、あるいは Hono + Vite + React
- DB: SQLite (better-sqlite3)
- ORM: Drizzle or Prisma
- xlsx: exceljs
- 配布: `npm run start` で localhost:3000 起動。LAN 共有
- 利点: 単一プロセス、動的 UI、エコシステム豊富
- 懸念: Next.js はやや重厚。SvelteKit or Hono の方がローカル向き

### 候補 B: Rust + embedded web server（推奨度: 中）

- ランタイム: Rust 1.94（インストール済み）
- フレームワーク: Axum + SQLite (sqlx or rusqlite)
- フロント: Vite + React/Svelte を静的配信
- xlsx: rust_xlsxwriter
- 利点: 単一バイナリ配布、起動高速、堅牢
- 懸念: 開発速度が Node より遅い

### 候補 C: Python FastAPI（推奨度: 低〜中）

- FastAPI + SQLite + SQLAlchemy
- フロント: 別途 Vite
- xlsx: openpyxl

### 共通推奨

- DB: SQLite（ファイル 1 個で完結、バックアップ容易）
- パッケージング: ローカル起動スクリプト `./start.sh` を用意
- ブラウザ: Safari / Chrome 最新、iPad Safari を主要ターゲット

### architect が確定すべき事項

- フレームワーク 1 つへの絞り込み
- 認証セッション方式（Cookie/JWT/ローカル固定トークン）
- LAN 公開時の IP / ポート / 証明書戦略
- ビルド成果物の配布形態
